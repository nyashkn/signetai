# Replay Harness Audit: TypeScript Test Corpus and Rust Shadow Internal-State Parity

**Summary.** Current route replay coverage is **310 / 319 public TS daemon routes (97.2%)** according to the generated parity map, but endpoint-specific shadow rules cover only **55 rule keys** (51 route-map `parityRule: true` entries), and the live shadow comparator compares **HTTP status + JSON response bodies only**. The current contract replay suite is strong as a Rust daemon regression suite, including many ad hoc DB assertions, but it is not yet a generated TS-test replay corpus and it has no uniform `(request, seed state, expected response, expected internal state)` fixture format.

## Header snapshot

- **Route coverage today:** 310 routes replay-proven, 9 routes mounted-shallow; no `missing` routes in `route-parity.json` (`platform/daemon-rs/contracts/route-parity.json:10`).
- **Coverage percent:** 310 / 319 = **97.2% route replay presence**; **2.8% shallow**.
- **Endpoint-specific shadow rules:** 55 entries in `parity-rules.json`; default rule compares JSON while ignoring top-level `updatedAt`/`createdAt` (`platform/daemon-rs/contracts/parity-rules.json:4`).
- **Biggest route-map gap:** auth methods/login/SSO/SAML/API-key CRUD are mounted but not replayed and have no parity rule (`platform/daemon-rs/contracts/route-parity.json:135`).
- **Biggest comparator gap:** `signet-shadow` records status and JSON-body divergences, not headers, byte bodies, DB rows, files, queues, embeddings, graph state, provenance, or scoping (`platform/daemon-rs/crates/signet-shadow/src/compare.rs:81`).
- **Corpus plan:** add a generated replay fixture layer from the 167 TS `*.test.ts` files: AST inventory + record-mode instrumentation + normalized JSON/SQL fixtures.
- **Shadow plan:** add route-aware post-request internal snapshots to `signet-shadow`, starting with SQLite snapshots for memories, embeddings, ontology, pipeline jobs, sessions, documents/connectors, telemetry, and scoped metadata.

### 1.3 Contract replay harness behavior

### 1.5 Shadow comparator behavior

`signet-shadow` is a live proxy in front of TS primary and Rust shadow daemons (`platform/daemon-rs/crates/signet-shadow/src/main.rs:1`). Defaults are proxy `3849`, primary `3850`, shadow `3851` (`platform/daemon-rs/crates/signet-shadow/src/main.rs:37`). It forwards request headers except hop-by-hop headers (`platform/daemon-rs/crates/signet-shadow/src/main.rs:255`) and forwards the body to both daemons. Primary is blocking/user-facing; shadow is async fire-and-compare (`platform/daemon-rs/crates/signet-shadow/src/main.rs:120`, `platform/daemon-rs/crates/signet-shadow/src/main.rs:154`).

The response snapshot contains only `status`, `body_bytes`, parsed `body_json`, and `content_type` (`platform/daemon-rs/crates/signet-shadow/src/main.rs:239`). The comparator:

- Always compares status codes; status mismatches are critical by default and consult only `errorComparison.mustMatch` (`platform/daemon-rs/crates/signet-shadow/src/compare.rs:90`).
- Compares JSON bodies only when both responses parse as JSON (`platform/daemon-rs/crates/signet-shadow/src/compare.rs:116`).
- Recursively checks object keys, extra keys, arrays by index, and primitive values (`platform/daemon-rs/crates/signet-shadow/src/compare.rs:156`).
- Uses endpoint rules to ignore fields and mark deterministic fields critical; non-deterministic fields on a ruled endpoint are logged as expected, not ignored (`platform/daemon-rs/crates/signet-shadow/src/compare.rs:303`).
- Does **not** compare headers, content type, non-JSON response bytes, streaming bodies, latency thresholds, request side effects, DB state, filesystem state, or queue state.

Divergence entries contain timestamp, endpoint, primary/shadow status, primary/shadow latency, and a list of field-level divergences with stringified values (`platform/daemon-rs/crates/signet-shadow/src/divergence.rs:19`, `platform/daemon-rs/crates/signet-shadow/src/divergence.rs:29`). The analysis subcommand summarizes critical/expected counts and status mismatches from the JSONL log (`platform/daemon-rs/crates/signet-shadow/src/main.rs:298`).

## 2. Gaps

### 2.1 Route and behavior gaps

1. **Auth shallow routes:** `GET/POST /api/auth/api-keys`, `DELETE /api/auth/api-keys/:id`, `POST /api/auth/login`, `GET /api/auth/methods`, `GET /api/auth/sso/start`, `GET /api/auth/sso/callback`, `POST /api/auth/saml/acs`, `GET /api/auth/saml/start` are mounted but not replayed and have no parity rules (`platform/daemon-rs/contracts/route-parity.json:135`).
2. **Many replay-proven routes are status/shape smoke tests, not semantic state tests.** The catch-all `remaining_public_routes_have_contract_replay_coverage` hits many endpoints with empty/missing inputs and expected statuses (`platform/daemon-rs/crates/signet-daemon/tests/contract_replay.rs:9950`), but it cannot prove all TS branch behavior.
3. **Parity rules omit most route-map entries.** Only 51 route-map entries have `parityRule: true`; routes without rules fall back to whole JSON comparison with default timestamp ignores, or in shadow become critical for any non-ignored JSON mismatch.
4. **Non-JSON and streaming surfaces are weakly compared.** Contract replay manually checks some SSE content types and connected markers for logs/cross-agent/OS streams, but shadow does not diff streaming payloads or headers (`platform/daemon-rs/crates/signet-daemon/tests/contract_replay.rs:5436`, `platform/daemon-rs/crates/signet-daemon/tests/contract_replay.rs:5966`).
5. **Runtime-specific watchers/CLI/process behavior is hand-tested, not corpus-convertible yet.** Watcher sync/import/autocommit tests rely on real filesystem polling and git state (`platform/daemon-rs/crates/signet-daemon/tests/contract_replay.rs:8051`, `platform/daemon-rs/crates/signet-daemon/tests/contract_replay.rs:8114`, `platform/daemon-rs/crates/signet-daemon/tests/contract_replay.rs:8179`).

### 2.2 Internal-state gaps for shadow

The live shadow comparator has no visibility into internal state, so the following can diverge while HTTP responses match:

- **Scoping:** `memories.agent_id`, `visibility`, `scope`; entity/aspect/attribute/dependency `agent_id`; session summary/transcript agent scoping; auth token scoped claims.
- **Attribution/provenance:** `source_type`, `source_id`, `source_path`, `source_root`, `runtime_path`, `idempotency_key`, proposal evidence, source artifacts, memory artifacts, `created_by`, `updated_by`, `changed_by`, actor/session/request metadata.
- **Memory mutation state:** `is_deleted`, `deleted_at`, version, content hash, normalized content, history rows, dedupe merge rows, chunk-group entities, cold memory archives.
- **Embeddings/vector state:** `embeddings` metadata, dimensions, content hash, `chunk_text`, `vec_embeddings` presence/count, null-vector/model-drift health; vector values should be compared by hash/dim/norm/tolerance, not exact raw bytes when providers differ.
- **Ontology graph state:** `entities`, `relations`, `memory_entity_mentions`, `entity_aspects`, `entity_attributes`, `entity_aliases`, `entity_dependencies`, `ontology_proposals`, `epistemic_assertions`, claim version roots, supersession links, proposal/application statuses.
- **Pipeline/queue state:** `memory_jobs`, `summary_jobs`, `ingestion_jobs`, retry/lease/attempt/dead/completed fields, checkpoint extract cursors, source-index jobs.
- **Session/injection state:** `sessions`, `session_memories`, transcripts, summaries, checkpoints, bypass flags, runtime-path conflict state.
- **Connectors/sources/documents:** `connectors.cursor_json`, `settings_json`, document status/error/link rows, connector sync jobs, source snapshots/import jobs.
- **Telemetry/analytics/plugin/audit:** request counters, provider errors, memory search telemetry, plugin audit redaction, marketplace review ledgers.
- **Filesystem ledgers:** `agent.yaml`, generated AGENTS/workspace docs, marketplace/secrets/plugin JSON ledgers, generated notes, git commits.

## 3. Corpus plan: derive replay fixtures from the 167 TS tests

### 3.1 Goal

Create a generated corpus with stable tuples:

```text
(source TS test) -> (seed DB/files/env, HTTP request(s), expected response, expected DB/files/internal state)
```

Then make Rust `contract_replay.rs` load and execute those fixtures in addition to hand-written tests. The generated corpus should be deterministic, reviewable, and small enough to check in.

### 3.2 Proposed fixture layout

```text
platform/daemon-rs/contracts/replay-corpus/

  cases/
    <case-id>.json
    <case-id>.seed.sql
```

### 3.3 Generator approach

Implement a generator such as `scripts/generate-rust-replay-corpus.ts` with two passes.

**Pass A: static inventory.**

- Walk the 167 files (`find platform/daemon -name '*.test.ts'` currently returns 167).
- Parse with TypeScript/Bun AST and emit an inventory of `describe`/`test`/`it` names, imports, route strings, HTTP calls (`app.request`, `fetch`, local route helpers), DB setup (`db.prepare`, `runMigrations`, `initDbAccessor`), filesystem setup, env mocks, timers, and mocked providers.
- Classify cases:
  - `http-db`: direct route tests with seeded SQLite and stable response/state expectations.
  - `http-files`: route tests with workspace/filesystem expectations.
  - `state-only`: transaction/DB helper tests that can become internal-state fixtures but not HTTP replay yet.
  - `runtime-specific`: watcher/process/socket/timer/git/update tests requiring bespoke harness support.
  - `provider-mocked`: tests that need fixture HTTP servers or mocked `fetch` responses.

**Pass B: record mode.**

Static extraction will miss too much setup logic, so use execution instrumentation for high fidelity:

1. Run selected TS tests under `SIGNET_REPLAY_CAPTURE=1` with a preload module.
2. Patch/wrap:
   - Hono/app request entrypoints and global `fetch` to capture method, path, headers, body, response status/body/headers.
   - DB accessors (`initDbAccessor`, `getDbAccessor`, `runMigrations`) to mark a baseline snapshot after migrations and capture SQL/diff before each HTTP call.
   - time/UUID/random helpers where the TS runtime uses them, producing deterministic replacements.
   - temp workspace creation to archive fixture files with redaction.
3. Around each request, dump:
   - `seed.sql`: minimal deterministic SQL for rows that differ from an empty migrated DB before the request.
   - `expectedResponse`: response status/body/headers from TS.
   - `expectedInternalState`: minimal post-request DB/file diff for tables relevant to the request family.
4. Normalize volatile values: IDs can be mapped to placeholders, timestamps to `$NOW`/precision windows, secrets to `[REDACTED]`, absolute temp dirs to `$WORKSPACE`, and provider-specific embeddings to dimension/hash/norm comparators.
5. Review generated cases: fail generation if a test has unredacted secrets, raw absolute temp paths, unordered row sets without `orderBy`, or no request/state captured.

### 3.4 Feasible vs runtime-specific TS tests

**High-feasibility first wave.** Convert route/API tests with stable request/response semantics:

- `platform/daemon/src/routes/*.test.ts` (18 files): auth/api keys, database diagnostics, git sync, graphiq, marketplace/reviews/MCP analytics, memory routes/scope, misc/provider safety, pipeline models/agent, plugins, reflections, secrets, skills, sources.
- API-style root tests: `agent-id`, `analytics`, `auth/*`, `cross-agent`, `daemon-auth-guard-colocation`, `diagnostics`, `graphiq`, `hooks*`, `knowledge-*`, `marketplace-*`, `mcp/*`, `memory-*`, `session-*`, `sources-*`, `telemetry`, `version`, `update-route`.
- DB-backed helper tests that map directly to route side effects: `transactions.test.ts`, `source-artifact-graph.test.ts`, `session-memories.test.ts`, `structural-features.test.ts`, `task-scope.test.ts`, `temporal-expand.test.ts`.

**Medium-feasibility with fixture adapters.** Convert provider-mocked/pipeline tests after adding fake provider HTTP fixtures and deterministic worker drains:

- Pipeline extraction/decision/summary/retention/maintenance tests under `platform/daemon/src/pipeline`.
- Inference router/API tests with mocked fetch/command executors.
- Embedding/vector tests, once vector comparison can use dimensions/hash/norm and optional sqlite-vec rows.
- Discord/GitHub/source fetch tests, by capturing provider responses into local fixture servers.

**Runtime-specific or bespoke.** Keep as hand-written or add special harnesses before conversion:

- Watcher/file-sync/identity-sync/git autocommit/polling tests.
- Bind/socket/CORS/server lifecycle tests.
- Scheduler spawn/worker process tests.
- Update-system tests involving package resolution, desktop/systemd/launchd, and process exit.
- Event-loop responsiveness/single-flight timing tests.

### 3.5 Rust runner changes

Add a data-driven module to `contract_replay.rs` or a sibling test file:

1. Load `manifest.json`.
2. For each case, create a fresh temp `SIGNET_PATH`, write `agent.yaml` and file fixtures, start daemon and let migrations run, apply `seed.sql` directly to `memory/memories.db`, then send the request.
3. Compare response with the fixture comparator (reuse/port shadow compare rules where possible).
4. Query expected DB snapshots with stable ordering and redactions.
5. For multi-request TS tests, support `steps[]`, each with request/response/post-state.
6. Emit a per-case route list back into `route-parity.json` generation so route coverage is derived from fixtures, not only hand-maintained Rust test names.

## 4. Shadow deepening plan: compare internal state

### 4.1 Design

Extend `signet-shadow` from HTTP-response diffing to response + internal-state diffing. It should stay safe for live use, so internal snapshots must be opt-in:

```bash
signet-shadow \
  --primary-host http://127.0.0.1:3850 \
  --shadow-host http://127.0.0.1:3851 \
  --primary-signet-path /tmp/signet-ts \
  --shadow-signet-path /tmp/signet-rs \
  --internal-state memories,history,embeddings,ontology,jobs,sessions
```

Or environment equivalents:

```text
SIGNET_SHADOW_PRIMARY_PATH
SIGNET_SHADOW_SHADOW_PATH
SIGNET_SHADOW_INTERNAL_STATE
SIGNET_SHADOW_INTERNAL_RULES
```

### 4.2 Code locations

- `platform/daemon-rs/crates/signet-shadow/src/main.rs`
  - Add config fields for primary/shadow workspace or DB paths.
  - After shadow request completion, capture snapshots from both DBs before writing the divergence entry.
  - For mutating routes, wait for a bounded quiescence window or route-specific worker drain before snapshotting; default to immediate read-only snapshot.
  - Include request ID/correlation and internal snapshot status in logged entries.

- `platform/daemon-rs/crates/signet-shadow/src/compare.rs`
  - Add rule types for internal state, e.g. `internalState.tables`, per-route table lists, ignored columns, key columns, tolerances, redactions.
  - Add `compare_internal(endpoint, primary_snapshot, shadow_snapshot)` returning `Vec<Divergence>`.
  - Enforce currently-dead rule concepts where applicable: unordered arrays, numeric tolerances, timestamp precision.

- `platform/daemon-rs/crates/signet-shadow/src/divergence.rs`
  - Extend `Divergence` with optional `category`, `table`, `key`, `primary_json`, `shadow_json`, or keep compatibility by stringifying structured values.
  - Extend `DivergenceEntry` with `requestId`, `internalCompared`, `snapshotErrors`, and maybe `caseId` for corpus replay.

### 4.4 Normalization rules

- Always key rows by stable columns (`id`, `memory_id`, `entity_id`, `source_id`, `session_key`) and sort before comparison.
- Ignore or precision-normalize timestamps unless the route explicitly promises a timestamp field.
- Redact secret-like columns/JSON keys (`token`, `secret`, `password`, `apiKey`, `clientSecret`, env values).
- Compare embeddings by metadata first; compare vector bytes only under deterministic local fixture providers, otherwise compare `dimensions`, row presence, non-null/non-zero status, source links, and content hash.
- For generated IDs, support placeholder capture from response to DB assertions (`$response.id`, `$step[0].json.id`).
- For background jobs, compare terminal fields only after an explicit drain condition; otherwise compare queue enqueue intent (`job_type`, `payload`, `status in [...]`).
